In a previous post, I decided that I would accept patches using a public-inbox. One advantage of public-inboxes is that anyone can mirror them. There‚Äôs a few legal issues associated with accepting patches via public-inboxes and allowing people to mirror public-inboxes:

  1. By default, emails are copyrighted. In other words, I‚Äôll need permission to redistribute any emails that get sent to my public-inbox.
  2. Even if I had permission to redistribute the emails, others wouldn‚Äôt necessarily have permission to redistribute them further.
  3. Even if I had permission to do 1 and 2, I still wouldn‚Äôt necessarily have permission to incorporate emails that contain patches into upstream projects.

How others have solved similar issues

I‚Äôm certainly not the first person to run into these sorts of challenges. Let‚Äôs take a look at how others have solved similar issues:

Stack Overflow

Stack Overflow isn‚Äôt meant to be mirrored or contain patches per se, but readers are often encouraged to copy from posters‚Äô answers. To make sure that readers have legal permission to copy from answers, Stack Overflow first makes sure that posters agree to its Terms of Service. Stack Overflow does this using three different mechanisms. The first is by linking to the Terms at the bottom of every page. If you click on that link, you‚Äôll see that the first sentence is By using our products, you agree to our Terms of Service. The Terms go on to say:

This first mechanism is something that I would like to avoid. How could anyone expect someone to truly (not just legally) agree to those terms before they use Stack Overflow? Even if the first time you visited Stack Overflow was by going directly to the Terms of Service page, you still would still be using the site before you had fully read those terms. Sending an HTTP request for the Terms and interpreting the response counts as using the site (that might not be legally true, but it‚Äôs definitely literally true). In other words, you have to use the site to figure out what legal terms you‚Äôve agreed to by using the site. It‚Äôs a trap!

The second mechanism for ensuring agreement to the Terms is gating off certain functionality. While you can use Stack Overflow without an account, you need an account to do certain things. The registration page says By clicking ‚ÄòSign up‚Äô, you agree to our terms of service, privacy policy and cookie policy. This mechanism is something that I‚Äôm much more OK with. This mechanism tells users exactly what they‚Äôre agreeing to before they have an opportunity to agree to it.

The final mechanism for ensuring agreement is the simplest. To the right of every ‚ÄúPost Your Answer‚Äù button is the text By clicking ‚ÄòPost Your Answer‚Äô, you agree to our terms of service, privacy policy and cookie policy.

GitHub

GitHub is a little bit closer to my use case than Stack Overflow. Stack Overflow only accepts comments that might contain snippets of code. GitHub accepts both comments and pull requests.

Like Stack Overflow, GitHub starts by making sure that everyone agrees to its Terms of Service. Unlike Stack Overflow, GitHub doesn‚Äôt try to make its Terms automatically apply to everyone who visits the site. Instead, there‚Äôs some text at the bottom of the sign-up page that says By creating an account, you agree to the Terms of Service.

If that patch was sent to me, I would assume that foo.txt was available under the same license as the rest of the repo. In reality, most patches don‚Äôt contain enough information for us to determine what legal terms apply to them. Now, let‚Äôs imagine that this change was submitted as a pull request rather than a patch. Before you can submit a pull request, you have to push all of the pull request‚Äôs commits to a remote. Commits are snapshots which means that every commit contains (a reference to) every file that was being tracked at the time. In other words, before you make a pull request, you have to give the recipient a complete copy of your version of the project. That complete copy would (probably) contain a license for the changes.

Even if the license file wasn‚Äôt modified, it still isn‚Äôt clear to me what the patch‚Äôs license would be. When I receive a copyright license, I expect there to be a statement telling me what the license is. As it turns out, that isn‚Äôt always the case. I recently found out that there‚Äôs something known as an implied license. In other words, when someone sends me a patch, it‚Äôs possible that I‚Äôll get a license for that patch even if the email doesn‚Äôt mention licensing or permissions at all. The question is: what license would I get in this situation? I would want it to be the same as what the upstream project is using, but I don‚Äôt know how likely that would be.

The Git project itself

Git is developed on a mailing list. That mailing list is available as a public-inbox where people contribute to Git by submitting patches. This means that the Git project does something very similar to what I want to do.

From what I can tell, the Git project doesn‚Äôt explicitly get permission to distribute most mailing list posts. It‚Äôs possible that sending an email to git@vger.kernel.org gives the Git project an implied copyright license to distribute that email. That license probably wouldn‚Äôt give others permission to mirror the project‚Äôs public-inbox. Additionally, the earliest email that I found in the archive is from 2005, long before the public-inbox project was started. While it‚Äôs possible that in 2005, posting to the Git mailing list gave the project an implied license to distribute the messages in an archive, I doubt that that license would give others permission to redistribute Git‚Äôs public-inbox.

Initially, I thought about using the DCO for my projects. It wouldn‚Äôt be a complete solution, but it would offer some legal clarity for patches. Plus, it‚Äôs relatively well known among FOSS contributors. Unfortunately, the DCO doesn‚Äôt really work the way I want it to:

  • The DCO has contributors certify that they have the right to submit their contribution under an appropriate license. It doesn‚Äôt make them certify that their contribution is actually under an appropriate license.
  • The DCO only works for open-source licenses. Many of the files in my projects are dedicated to the public domain under üÖ≠üÑç. üÖ≠üÑç is a public domain dedication, not a license. If üÖ≠üÑç was a license, then it probably wouldn‚Äôt be an open-source one.
  • The DCO only works for [‚Ä¶]the open source license indicated in the file. I try to make many of the files in my projects indicate their license or public domain dedication, but there are some files where that simply isn‚Äôt the best option. I‚Äôm sure that it‚Äôs possible to embed licensing metadata in an image file, but I doubt that many people would know how to access that metadata. Plus, there‚Äôs definitely some file formats where you couldn‚Äôt add licensing metadata in a reasonable way.

Others

While every project and service is different, my research indicates that most projects fall into one or more of these categories:

  • projects that rely on implied licenses,
  • projects that let their centralized hosting platform handle it (by requiring pull requests or agreement to a ToS) or
  • projects that have some sort of CLA (like the DCO).
